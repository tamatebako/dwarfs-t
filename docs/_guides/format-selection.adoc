== Overview

DwarFS v0.16.0 supports two metadata serialization formats. This guide helps you choose the right format for your use case.

== Detailed Comparison

=== FlatBuffers (Modern Default)

==== Strengths ✅

* **Universal Portability**
  - Works on all platforms: Linux, macOS, Windows, FreeBSD
  - Works on all architectures: x86_64, aarch64, arm, ppc64, riscv64, s390x, loongarch64
  - No platform-specific code paths

* **Simple Dependencies**
  - Header-only FlatBuffers library (auto-fetched via CMake)
  - No complex build requirements
  - Fast compilation (no Folly overhead)

* **Static Linking Compatible**
  - Works with Tebako packaging
  - Works with fully static builds (Alpine, musl libc)
  - No ABI stability concerns

* **Excellent Performance**
  - Zero-copy access via memory mapping
  - Fast verification (<1ms for typical metadata)
  - Efficient random access

* **Forward/Backward Compatible**
  - Schema evolution built-in
  - Can add fields without breaking old readers
  - Graceful handling of unknown fields

==== Limitations ⚠️

* **Slightly Larger Size**
  - ~5-10% larger than Thrift Frozen2
  - Not significant for most use cases
  - Example: 100 MiB Thrift → 105-108 MiB FlatBuffers

* **New Format**
  - Cannot read pre-v0.16.0 DwarFS images
  - Requires both-formats build to migrate old images
  - Not tested in production as long as Thrift

==== Best For

* ✅ New DwarFS images
* ✅ Cross-platform deployments
* ✅ Static linking scenarios
* ✅ Tebako packaging
* ✅ Simplified build environments
* ✅ Windows, macOS development

=== Thrift Frozen2 (Legacy)

==== Strengths ✅

* **Maximum Size Efficiency**
  - Bit-packed Frozen2 structures
  - Smallest possible metadata size
  - Critical for bandwidth-constrained scenarios

* **Battle-Tested**
  - Proven in production since DwarFS inception
  - Handles edge cases reliably
  - Extensive compatibility history

* **Backward Compatible**
  - Reads all DwarFS versions (v0.2.0+)
  - Essential for reading existing images
  - Smooth transition support

* **Zero-Copy Performance**
  - Memory-mapped Frozen2 layouts
  - Instant access to metadata
  - No deserialization overhead

==== Limitations ⚠️

* **Complex Dependencies**
  - Requires Facebook Folly (large C++ library)
  - Requires fbthrift (Thrift compiler + runtime)
  - Long build times (10-15 minutes extra)

* **Platform Limitations**
  - Not available on Windows
  - Not available on older macOS versions
  - Not available on some architectures
  - MSys2/MinGW incompatible

* **Static Linking Issues**
  - Folly has ABI stability concerns
  - Difficult to link statically
  - Incompatible with Tebako

* **Build Complexity**
  - Requires specific compiler versions
  - Platform-specific build issues common
  - Dependency management challenging

==== Best For

* ✅ Reading existing Thrift images
* ✅ Absolute minimum size requirements
* ✅ Linux-only deployments
* ✅ Maintaining legacy workflows

== Use Case Scenarios

=== Scenario 1: New DwarFS User

**Situation**: Starting fresh with DwarFS

**Recommendation**: ✅ **FlatBuffers-Only**

[source,bash]
----
cmake -B build -GNinja \
  -DDWARFS_WITH_FLATBUFFERS=ON \
  -DDWARFS_WITH_THRIFT=OFF
----

**Rationale**:
- Simplest setup
- Works everywhere
- Default format
- No legacy concerns

=== Scenario 2: Existing Thrift Images

**Situation**: Have existing DwarFS images in Thrift format

**Recommendation**: ✅ **Both Formats**

[source,bash]
----
cmake -B build -GNinja \
  -DDWARFS_WITH_FLATBUFFERS=ON \
  -DDWARFS_WITH_THRIFT=ON
----

**Workflow**:
1. Build with both formats enabled
2. Read old Thrift images
3. Create new images in FlatBuffers format
4. Gradually migrate to FlatBuffers-only builds

**Migration**:
[source,bash]
----
# Extract old Thrift image
./build/dwarfsextract -i old.dwarfs -o /tmp/extracted

# Create new FlatBuffers image
./build/mkdwarfs -i /tmp/extracted -o new.dwarfs

# Verify
./build/dwarfsck -i new.dwarfs --checksum
----

=== Scenario 3: Size-Critical Deployment

**Situation**: Network bandwidth expensive, storage limited

**Recommendation**: ⚠️ **Thrift-Only** (if platform supports)

[source,bash]
----
cmake -B build -GNinja \
  -DDWARFS_WITH_FLATBUFFERS=OFF \
  -DDWARFS_WITH_THRIFT=ON
----

**Consideration**: 5-10% size increase with FlatBuffers usually acceptable given:
- Better compression algorithms (zstd) have larger impact
- Metadata is small fraction of total image size
- Portability benefits outweigh size cost

=== Scenario 4: Static Linking / Tebako

**Situation**: Need static linking or Tebako packaging

**Requirement**: ✅ **FlatBuffers-Only** (mandatory)

[source,bash]
----
cmake -B build -GNinja \
  -DDWARFS_WITH_FLATBUFFERS=ON \
  -DDWARFS_WITH_THRIFT=OFF \
  -DTEBAKO_BUILD=ON
----

**Reason**: Thrift/Folly incompatible with static linking

=== Scenario 5: CI/CD Pipeline

**Situation**: Automated builds across platforms

**Recommendation**: ✅ **FlatBuffers-Only**

**Benefits**:
- Consistent builds across all platforms
- No platform-specific conditionals
- Faster build times
- Simpler dependency management

== Runtime Behavior

=== Format Detection

DwarFS automatically detects the format when reading images:

[source]
----
Image Header
    │
    ├─► Check for FlatBuffers magic bytes "DFBF"
    │        │
    │        ├─► Found: Use FlatBuffers reader
    │        └─► Not found: Fall back to Thrift
    │
    └─► If Thrift disabled at build time: Error
----

=== Format Selection (mkdwarfs)

When creating images, format is selected at **build time**:

* **FlatBuffers-only build**: Always creates FlatBuffers images
* **Thrift-only build**: Always creates Thrift images  
* **Both-formats build**: Prefers FlatBuffers (new default)

== Migration Timeline Recommendation

=== For Existing Users

**Phase 1** (v0.16.0 release):
- Use both-formats builds
- Keep reading old Thrift images
- Start creating new FlatBuffers images

**Phase 2** (3-6 months):
- Continue both-formats
- Gradually migrate critical images
- Test FlatBuffers in production

**Phase 3** (v0.17.0+):
- Switch to FlatBuffers-only

- Thrift becomes fully optional

== Frequently Asked Questions

**Q: Can I convert Thrift images to FlatBuffers?**

A: Yes, via extract + recreate:
[source,bash]
----
dwarfsextract -i old.dwarfs -o /tmp/data
mkdwarfs -i /tmp/data -o new.dwarfs
----

**Q: What happens if I try to read FlatBuffers image with Thrift-only build?**

A: Error: "Unsupported metadata format" (cannot read without FlatBuffers support)

**Q: Can I read both formats with same binary?**

A: Yes, if built with both formats enabled (`DWARFS_WITH_FLATBUFFERS=ON` + `DWARFS_WITH_THRIFT=ON`)

**Q: Which format is faster?**

A: Effectively equal - both use zero-copy memory mapping. Differences <1% in practice.

**Q: Should I care about 5% size increase?**

A: Usually no. Better compression algorithms have 10-100x larger impact on total image size.

== Recommendations Summary

✅ **Use FlatBuffers** if you:
- Are starting new
- Need cross-platform builds
- Want simple dependencies
- Need static linking
- Use Tebako
- Value build simplicity

⚠️ **Use Thrift** if you:
- Have existing Thrift images
- Absolutely need smallest size
- Linux-only deployment
- Already have Folly/Thrift infrastructure

🔄 **Use Both Formats** if you:
- Migrating from Thrift
- Need to read old images
- Transition period
- Testing both formats
